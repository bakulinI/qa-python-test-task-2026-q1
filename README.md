# QA Python — Тестовое задание

## Описание

Тестовое задание для позиции **QA Python Engineer**.  
Необходимо написать автотесты для REST API сервиса **Swagger Petstore**.

Ожидаемое время выполнения: **2-3 дня** с момента получения задания.

---

## Установка и запуск сервиса

### Предварительные требования

- Установленный [Docker](https://docs.docker.com/get-docker/)
  и [Docker Compose](https://docs.docker.com/compose/install/)

### Запуск

Из корня проекта выполнить:

```bash
docker compose up -d
```

Эта команда поднимет контейнер с сервисом **Swagger Petstore** на базе образа `swaggerapi/petstore3`.

После запуска будут доступны:

| Ресурс         | URL                          | Описание                                                                    |
|----------------|------------------------------|-----------------------------------------------------------------------------|
| **Swagger UI** | http://localhost:8080        | Интерактивная документация API (можно отправлять запросы прямо из браузера) |
| **API**        | http://localhost:8080/api/v3 | Базовый URL для всех запросов к API                                         |

Проверить работоспособность:

```bash
curl http://localhost:8080/api/v3/pet/findByStatus?status=available
```

Остановка сервиса:

```bash
docker compose down
```

## Задание

Написать набор автотестов для REST API Swagger Petstore. Полный список эндпоинтов и модели данных можно посмотреть в 
**Swagger UI** после запуска сервиса.

### Покрытие эндпоинтов

Ожидается покрытие тестами **всех основных эндпоинтов** сервиса:

#### Pet (`/pet`)

| Метод    | Эндпоинт            | Описание           |
|----------|---------------------|--------------------|
| `POST`   | `/pet`              | Создание питомца   |
| `PUT`    | `/pet`              | Обновление питомца |
| `GET`    | `/pet/findByStatus` | Поиск по статусу   |
| `GET`    | `/pet/findByTags`   | Поиск по тегам     |
| `GET`    | `/pet/{petId}`      | Получение по ID    |
| `DELETE` | `/pet/{petId}`      | Удаление питомца   |

#### Store (`/store`)

| Метод    | Эндпоинт                 | Описание               |
|----------|--------------------------|------------------------|
| `GET`    | `/store/inventory`       | Получение инвентаря    |
| `POST`   | `/store/order`           | Создание заказа        |
| `GET`    | `/store/order/{orderId}` | Получение заказа по ID |
| `DELETE` | `/store/order/{orderId}` | Удаление заказа        |

#### User (`/user`)

| Метод    | Эндпоинт               | Описание                      |
|----------|------------------------|-------------------------------|
| `POST`   | `/user`                | Создание пользователя         |
| `POST`   | `/user/createWithList` | Создание списка пользователей |
| `GET`    | `/user/login`          | Логин                         |
| `GET`    | `/user/logout`         | Логаут                        |
| `GET`    | `/user/{username}`     | Получение пользователя        |
| `PUT`    | `/user/{username}`     | Обновление пользователя       |
| `DELETE` | `/user/{username}`     | Удаление пользователя         |

### Сценарии тестирования

Для каждого эндпоинта ожидаются **позитивные и негативные** сценарии, а также **сквозные (E2E) цепочки**:

#### Позитивные сценарии

Корректные запросы с валидными данными. Проверяется:

- HTTP-код ответа
- Структура и содержимое тела ответа

#### Негативные сценарии

Запросы с невалидными данными. Примеры:

- Наличие негативных тестов для определенных эндпоинтов, их количество и содержание на ваше усмотрение. Ожидается, что
  такие тесты в целом будут.

#### E2E-сценарии

Сквозные цепочки, проверяющие связанную работу нескольких эндпоинтов.
Минимум, который ожидается (больше - на ваше усмотрение):

1. **Жизненный цикл питомца:**  
   Создать питомца → Получить по ID → Обновить → Проверить обновление → Удалить → Убедиться, что удалён

2. **Поиск питомцев:**  
   Создать несколько питомцев с одинаковым статусом → Найти по статусу → Убедиться, что все созданные питомцы
   присутствуют в результатах

3. **Полный пользовательский сценарий:**  
   Создать пользователя → Логин → Создать заказ → Получить заказ → Удалить заказ → Удалить пользователя

#### Параллельное выполнение тестов и асинхронность

Продемонстрировать параллельное выполнение тестов.

Продемонстрировать использование преимуществ асинхронности, например:

- Параллельное создание нескольких сущностей через `asyncio.gather` / `asyncio.TaskGroup`
- Параллельная проверка нескольких условий
- Параллельная очистка данных после теста

### Требования к реализации

- **Python 3.11+**
- **pytest** в качестве тестового фреймворка
- **Асинхронные тесты** с использованием `async/await`
- **Allure** для формирования отчётов, селекции и структурирования тестов (Использование step'ов, id, тегов,
  формирование отчета в папке `allure-results`)
- **Настроено управление зависимостями**
- Тесты должны быть **независимы** друг от друга — каждый тест сам создаёт и очищает свои тестовые данные
- Выбор HTTP-клиента и остальных библиотек — **на усмотрение кандидата**

### Структура проекта

Кандидат самостоятельно определяет структуру проекта.

Оценивается:

- Логичное разделение тестов по модулям
- Выделение вспомогательного кода (фикстуры, хелперы, модели, API-клиент)
- Большим плюсом будет генерация API-клиента на основе OpenAPI-спецификации, но это не обязательно
- Читаемость и поддерживаемость кода
- Конфигурация (base URL, таймауты и т.д.) вынесена в отдельное место
- Наличие простейшего логирования
- Использование allure для структурирования тестов и генерации отчётов

### Известный дефект Swagger Petstore (образ `swaggerapi/petstore3:1.0.27`)

В процессе выполнения задания был обнаружен стабильный дефект сервиса Swagger Petstore: при обращении к эндпоинту `POST /pet` сервер возвращает **500 Internal Server Error** даже для валидного JSON, сгенерированного самим Swagger UI.

Пример запроса (из Swagger UI):

```bash
curl -X 'POST' \
  'http://localhost:8080/api/v3/pet' \
  -H 'accept: application/json' \
  -H 'Content-Type: application/json' \
  -d '{
  "id": 10,
  "name": "doggiech",
  "category": {
    "id": 1,
    "name": "Dogs"
  },
  "photoUrls": [
    "string"
  ],
  "tags": [
    {
      "id": 0,
      "name": "string"
    }
  ],
  "status": "available"
}'
```

Пример ответа сервера:

```json
{
  "code": 500,
  "message": "There was an error processing your request. It has been logged (ID: b258e24fd37053f5)"
}
```

Этот же результат (HTTP 500) воспроизводится:

- при отправке аналогичного запроса из Swagger UI;
- при прямом HTTP-запросе через `httpx`/`curl` в обход тестового фреймворка;
- в автотестах, использующих `POST /pet`.

Поэтому часть тестов, зависящих от `POST /pet` (создание питомцев и цепочки E2E, использующие этот шаг), падает именно из-за дефекта сервиса, а не логики тестов. В соответствующих тестах в коде добавлены явные комментарии с указанием на эту особенность образа `swaggerapi/petstore3:1.0.27`. Остальные тесты (не зависящие от `POST /pet`) проходят успешно и демонстрируют как позитивные, так и негативные сценарии, а также асинхронность и параллельный запуск. 
Командой allure generate allure-results -o allure-report --clean репорт создать
